# 批判会议记录

## 会议状态：进行中

## Team Lead 批判意见

### 🔴 高风险问题

#### 1. CLI 进程管理风险
**问题描述**:
- `spawn('claude', ...)` 依赖本地 Claude CLI 安装
- 进程崩溃后可能导致资源泄漏（文件句柄、内存）
- 多个 CLI 进程同时运行时可能超出系统限制

**风险等级**: 🔴 高
**建议替代方案**:
1. 添加 CLI 版本检测和健康检查
2. 实现进程池管理，限制最大并发数
3. 添加进程崩溃后的自动恢复机制

#### 2. 文件通信的并发安全
**问题描述**:
- 多个 Team 同时读写 inbox/outbox 目录
- 文件锁竞争可能导致死锁
- 读写冲突可能导致数据损坏

**风险等级**: 🔴 高
**建议替代方案**:
1. 使用文件锁（如 `proper-lockfile`）
2. 改用 SQLite 作为通信介质
3. 实现消息队列（如 Redis）

### 🟡 中风险问题

#### 3. SSE 推送的可靠性
**问题描述**:
- 客户端断线后消息丢失
- 多标签页同时打开时状态不一致
- 消息顺序可能错乱

**风险等级**: 🟡 中
**建议替代方案**:
1. 添加消息队列和重连机制
2. 使用消息 ID 去重和排序
3. 实现客户端状态同步

#### 4. 错误处理与恢复
**问题描述**:
- CLI 进程崩溃后 Lead 无法感知
- 超时后如何优雅退出
- 部分失败时如何降级

**风险等级**: 🟡 中
**建议替代方案**:
1. 实现完善的错误传播机制
2. 添加超时后的优雅退出策略
3. 实现降级处理（如跳过失败的 Team）

### 🟢 低风险问题

#### 5. 测试覆盖率
**问题描述**:
- CLI 进程难以 mock
- 集成测试依赖真实环境
- E2E 测试成本高

**风险等级**: 🟢 低
**建议替代方案**:
1. 使用录制的 CLI 输出作为 fixture
2. 实现 CLI 的抽象层，便于 mock
3. 分层测试策略（单元 → 集成 → E2E）
