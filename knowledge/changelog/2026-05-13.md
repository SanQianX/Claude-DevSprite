# 变更记录 - 仪表盘统计合并与审查操作刷新修复

## 提交信息

- **提交类型**: `fix` - 修复
- **提交消息**: `fix(dashboard): combine task+review stats, fix review operations refresh tasks`
- **提交日期**: 2026-05-13

## 变更概要

本次提交修复了 Web Dashboard 中项目计划/进度统计显示不准确，以及执行审查（approve/fix）操作后任务列表不更新的两个问题。同时，对分析管道（Pipeline）的配置机制进行了重构和增强。

## 详细变更

### 1. 修复仪表盘统计显示与数据刷新

**变更文件**:
- `web/src/views/DashboardView.vue`
- `web/src/stores/dashboard.ts`
- `tasks/bugfix/task-stats-not-updating/` (新增文档)

**变更详情**:

- **统计计算修正 (DashboardView.vue)**:
  - 将进度统计中的“完成”数分母从 `tasks.length` 修正为 `completedCount + inProgressCount + backlogCount`。这确保了总任务数能够包含所有状态的任务（待开发、进行中、已完成），解决了因任务状态不同步而导致的百分比计算错误问题。
  - 细节文本从“X 个任务”优化为“X 项”。

- **审查操作后刷新任务 (dashboard.ts)**:
  - 在 `approveReview` 和 `fixReview` 两个 action 函数的末尾，添加了 `await fetchTasks(projectName)` 调用。这解决了在用户批准或修复审查后，关联的任务列表（如从代码审查中创建的任务）不会自动刷新的 Bug。

- **新增 Bug 案例文档**:
  - 新增了 `tasks/bugfix/task-stats-not-updating/` 目录，包含完整的 Bug 修复流程文档（UI 分析、根因分析、总结），详细记录了问题的发现和解决方案。

**根本原因**:
1.  **统计不准确**: 仪表盘统计的计算使用了未反映所有任务状态的数组长度作为总数。
2.  **数据不同步**: 审查操作的后端处理流程中，可能创建或更新了任务数据，但前端状态管理（Store）在操作完成后未主动触发任务数据的重新获取，导致 UI 停留在旧数据状态。

### 2. 分析管道配置重构

**变更文件**:
- `src/analyzer/pipeline.ts`
- `src/analyzer/types.ts`
- `src/analyzer/aiProvider.ts`

**变更详情**:

- **新增配置类型 (types.ts)**: 定义了 `AnalysisConfig` 接口，包含 `maxRetries` 和 `retryBaseDelayMs` 字段，用于管理 AI 调用的重试策略。
- **管道配置注入 (pipeline.ts)**: `AnalysisPipeline` 构造函数接收可选的 `aiConfig`，并在内部创建 `AnalysisConfig` 实例。这实现了配置的依赖注入，使管道行为更可预测和可测试。
- **注释优化 (aiProvider.ts)**: 删除了冗长的多行注释块，并更新了模块级注释，使其更简洁，并与项目文档（`FUNCTIONAL-LOGIC-ANALYSIS`）的描述保持一致。

### 3. 配置与路径标准化

**变更文件**:
- `src/config.ts`
- `src/daemon.ts`
- `src/analyzer/contextBuilder.ts`

**变更详情**:

- **配置项增强与文档化 (config.ts)**: 为 `ProjectDiscoveryConfig` 和 `ServerConfig` 接口添加了详细的 JSDoc 注释，提升了代码可维护性。
- **路径标准化 (daemon.ts, config.ts)**: 将用户数据目录从 `~/.claude/claude-dev-sprite/` 统一为 `~/.claude-dev-sprite/`。这是之前一次修复的延续，确保所有路径一致。
- **知识库目录容错 (contextBuilder.ts)**: `loadRelevantKnowledge` 方法现在优先从配置读取知识库目录名，并提供了默认值 `‘knowledge’`，增强了健壮性。

### 4. 检测器文档增强

**变更文件**:
- `src/detectors/detectorFactory.ts`
- `src/detectors/index.ts`

**变更详情**:

- **架构位置说明 (detectorFactory.ts)**: 在模块注释中明确了该模块在系统架构中属于“事件源 (Event Sources)”层，并作为“提交检测 (Detectors)”子系统的核心。
- **回退机制摘要 (index.ts)**: 在 `CommitDetectorManager` 的注释中，清晰地列出了 Git Hook、.git 目录监控、Reflog 轮询这三级检测策略的回退顺序。

### 5. 其他变更

- **测试文件清理**: 删除了 `test-task-status-result.png` 和 `test-task-sync-result.png` 两个测试截图文件。
- **任务清单更新**: `tasks/bugfix/README.md` 中新增了 `task-stats-not-updating` 的条目。

## 影响评估

- **功能修复**: 核心影响是修复了 Dashboard 的任务进度统计和审查操作后的数据同步问题，提升了用户界面的准确性和交互的可靠性。
- **代码质量**: 通过重构管道配置、标准化路径和增强文档，提升了代码的可读性、可维护性和鲁棒性。
- **知识沉淀**: 新增的 Bug 修复案例文档为项目留下了宝贵的故障排查知识。
